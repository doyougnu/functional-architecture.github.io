Functional Software Architecture

Functional programming in the large

Functional Software Architecture refers to methods of construction and structure of large and long-lived software projects that are implemented in functional languages and released to real users, typically in industry.

Simplicity

Software design is fundamentally a human endeavour. Simple systems allow for more reasoning bandwith between the software and its designer.

Problem domains are complex enough. The solutions we build should be as simple as possible.

Models, composition, better abstractions

Domain Insight

Software development is about running software just as much as it is about gaining insight into a domain. We think that a rigorous, formal, math-inspired approach to software development supports the entire lifecycle of a project, including requirements engineering and acceptance.

Maintainability

Requirements and environments change constantly and software has to follow suit. Therefore maintainability and malleability are not just two goals among many others. Maintainability and malleability fundamentally enable the software creation process.

Correctness

We value software being correct both in the small and in the large.

If we want to judge whether our software is correct, we have to explicitly state what it means for the software to be correct first. The I-know-it-when-I-see-it approach to correctness doesn’t scale to large systems.

Performance

Software has to be both correct and fast. If software were not fast, computers would be obsolete.

Principles

We strive for the values described above. We do so by following this set of principles.

Immutablility

Object-oriented and procedural programming treat all data as a combination of two things: a place (in memory) and a value that occupies this place. In functional programming we omit the first concept and talk about immutable values directly. This makes programming way simpler because it frees the programmer from thinking in terms of resource and state management and allows her to think in terms of their domain.

These advantages carry over from functional programming to functional software architecture. We avoid mutable state no matter the size of the structure.

Learn more on Immutablility

Pure Functions

Functional software architects operate with pure functions. A pure function transforms immutable values without performing any side effects.

Learn more on Pure Functions

Everything as a Value

Functional programming languages allow functionality to be treated like any other value. We say that functions are first-class. In functional software architecture we build on that foundation by trying to treat other abstract concepts as values: lenses as first-class property accessors, components as first-class user interface building blocks, behaviours as first-class time-varying values …

Learn more on Values

Composition and Closure

In functional software architecture we like to combine small software structures to form larger structures. These larger structures are in some abstract sense no different from the smaller ones; indeed we can combine these larger structures again using the same techniques (“Closure Principle”).

Lots of our systems provide so called combinator models. These consist of a set of primitive structures and a set of glue devices – combinators – that adhere to the closure principle.

Airtight Abstractions

Abstraction is the sharpest weapon of reason. Functional software architects welcome abstraction as a tool for coping with complexity. They erect abstraction barriers between different parts of a program to decouple these parts and make them easy to reason about in isolation.

Abstractions must be free of leaks. We therefore need our abstractions to be described rigorously and in a formal manner. We tend to take to mathematics for abstractions that stood the test of time.

Learn more on Abstraction

Architecture as Code

Architecture activities usually yield artifacts separate from the working software, such as diagrams or descriptions in prose. These separate documents tend to get out of sync with the software system over time. In functional software architecture we use diagrams and descriptions as supporting documentation, but the source of truth is always to be found in the code. This neccessitates that our programming languages are powerful enough to express architectural structures.

Composable GUI libraries

Facebook’s React popularized the component model of user interface programming. Functional programming languages allow to improve on that model by treating components as composable first-class user interfaces. Functional UI libraries provide a set of primitive components and a set of UI combinators, which let you build sophisticated graphical user interfaces without cognitive overhead.

Denotational Design

Denotational Design is a software design methodology which tries to extract the essence of a domain’s problem and describe it formally in machine-checkable code. Denotational design affords software designers to be absolutely precise in what they want to achieve before they talk about how they plan to achieve it. Denotational Design informs both the use and the implementation of a unit of software without coupling them. Denotational Design is therefore a methodology to build airtight abstractions.

Frequently Asked Questions

Is there a class of domains where Functional Software Architecture works exceptionally well? Is there a class of domains where Functional Software Architecture fails?

No

External systems usually don't follow FSA principles. How do you interface with them?

Spec out their behaviour, making time (or state transitions) explicit. Ask for guarantees that the external service provides.

I want to [do X]. What techniques or libraries should I use?
  • I want to build a web frontend Elm architecture, Model-View-Update, reacl
  • I want to build a web backend Handlers as functions from request to response (Clojure ring, OCaml Dream ...)
  • I need to access a RDBMS ...
I need to serialize objects to send them back and forth over a channel

Is Domain-Driven Design at odds with functional modelling?

It depends.

Can I do functional software architecture in non-functional languages like Java, Python or C++?

Object-oriented and procedural languages usually lack the linguistic devices required to express the ideas of functional software architecture. Models expressed in these languages are usually shallow, you will struggle to describe your architecture in code, and place-oriented (von Neumann) programming leads to high coupling between building blocks. In addition, while it is often possible to program with immutable values in non-functional languages – be it by sheer self-discipline –, the ecosystem usually doesn’t follow suit.

Trying to do functional software architecture without functional languages requires you to go to such great lengths that it’s generally not worth the effort.
